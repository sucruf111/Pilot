Division of labor

Two models, two jobs

Pilot splits development the way a good team does: the expensive, careful thinker makes the decisions; the fast builder writes the code. The lead owns what and whether it's right. Codex owns how. Both seats are configurable — lead with Fable 5 or Opus 4.8, and route routine reviews to a cheaper model.

Claude Fable 5 team lead

Runs inside Claude Code. Never writes application code.

  • Interviews you and writes the product & design briefs
  • Designs the system: tokens, components, roadmap
  • Writes one precise work order per task, just in time
  • Reviews every diff against the spec before it merges

Codex CLI coding agent

Dispatched headlessly with codex exec, one branch per task.

  • Implements the work order — code, tests, assets
  • Generates the visual concepts you choose between
  • Fixes what the free gate or the review flags
  • Decides implementation details itself — no micromanaging
The pipeline

Four phases, two human gates

All state lives in files on disk — close the session anytime, /pilot resumes exactly where it stopped. Works on new and existing projects: for an existing repo, an adoption phase audits the codebase, derives the design system and tokens from what's actually there, and grandfathers old violations — old code never blocks new work.

A

Adopt an existing project optional

For repos that already have code: Fable surveys the stack, reverse-engineers the briefs and de-facto design tokens from the dominant patterns, seeds the lessons file with the house style, and builds the roadmap from where the project stands.

→ briefs, tokens & roadmap derived from the codebase
0

Goal iteration

Fable interviews you: what the product is, who it's for, what's essential versus later. Iterates until every scope decision is defensible.

→ docs/PRODUCT_BRIEF.md
1

Brand & design human gate

Fable writes a design brief with 2–3 visual directions; Codex builds the concepts. Everything stops until you pick one.

→ docs/DESIGN_BRIEF.md · docs/design/concepts/
2

Design system & roadmap

The highest-leverage pass: a design system document, a machine-readable design-tokens.json, and a roadmap of vertical slices — foundation task first, so every later screen composes checked components. Includes a cost estimate before anything is built.

→ DESIGN_SYSTEM.md · design-tokens.json · ROADMAP.md
3

Autopilot

Task by task: spec → implement → mechanical gate → review → merge. Interrupts you only for gated decisions, repeated failures, or roadmap changes.

→ specs/ · reviews/ · LESSONS.md · checked-off ROADMAP.md
Phase 3, per task

The autopilot cycle

A failing build never reaches (or bills) the reviewer — mechanical checks run first, for free.

Fable  writes a grounded spec from the actual codebase      ~$0.50
  │
Codex  implements on branch pilot/<task>
  │
gate   build · tests · token-lint · file-boundary check     $0
  │        └─ fails? → straight back to Codex, twice max
Fable  reviews the diff against spec + design system       ~$0.35
  │        └─ request_changes? → findings back to Codex
merge  PR created · roadmap ✓ · lesson recorded → next task
Why it stays consistent & cheap

Design principles

Consistency by construction

Every color and spacing value must come from design-tokens.json — a lint script rejects violations. A $0 grep beats a $0.50 model check.

Just-in-time specs

Task detail is written when the task is dispatched, grounded in the real codebase — never upfront, never stale.

Vertical slices

Data + UI + tests per task, not page-by-page. Independently reviewable, independently shippable.

Free gate before paid review

Build, tests and lint run before any model reviews the diff. Roughly half of all fix rounds cost nothing.

Diffs, not repos

The reviewer reads the change against the spec — never the whole codebase. Cached context does the rest.

A memory that compounds

Every review appends lessons to LESSONS.md, which rides in every future spec. The pipeline gets better as it runs.

Get started

Install in one copy

Requires Claude Code and the Codex CLI, inside a git repository.

# into one project
$ git clone https://github.com/sucruf111/Pilot
$ mkdir -p your-app/.claude/skills
$ cp -R Pilot/.claude/skills/pilot your-app/.claude/skills/pilot

# — or available in every project
$ cp -R Pilot/.claude/skills/pilot ~/.claude/skills/pilot

# then, inside a Claude Code session in your project:
> /pilot

Pick your models & effort

Every role in the pipeline is configurable per project via docs/pilot/config.json — which Claude leads, which model reviews, and how hard each one thinks. You are never locked to one model.

SettingOptionsGuidance
lead.model fable-5 · opus-4.8 Fable 5 for the hardest, most ambiguous projects; Opus 4.8 at half the price is excellent for most builds. Pilot checks your running session model and asks you to switch on mismatch.
lead.*_effort medium · high · xhigh · max Set separately for planning and review — xhigh for the roadmap/design pass, high for reviews is the sweet spot.
review.model inherit · opus · sonnet · haiku Run routine diff reviews on a cheaper model in a subagent — blockers and architectural questions escalate back to the lead. The biggest cost lever in Phase 3.
codex.* any Codex model · reasoning_effort up to xhigh Applied to every codex exec dispatch. 
// docs/pilot/config.json — created at preflight, edit anytime
{
  "lead":   { "model": "opus-4.8", "planning_effort": "xhigh", "review_effort": "high" },
  "review": { "model": "sonnet",   "escalate_blockers_to_lead": true },
  "codex":  { "model": "gpt-5.5",  "reasoning_effort": "xhigh" }
}
// ↑ example: Opus 4.8 leads, Sonnet handles routine reviews, blockers escalate
